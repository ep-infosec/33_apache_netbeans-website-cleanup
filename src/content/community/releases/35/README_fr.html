Introduction

Merci d'utiliser l'EDI NetBeansTM 3.5, un environnement de développement intégré (EDI) modulaire, basé sur les standards, écrit en JavaTM. Le projet NetBeans consisten en:

  • un EDI open source écrit en Java.
  • une plateforme pour applications, qui peut être utilisé en tant que framework générique pour concevoir n'importe quel type d'application.

En savoir plus à propos du projet NetBeans ...

Vous pouvez trouver la dernière version de ce document à l'adresse suivante : https://netbeans.org/community/releases/35/README_fr.html.

Ce Qui A Changé

Changements depuis la version 3.4.1

Le travail effectué sur la Release 3.5 du EDI NetBeans s'est entièrement focalisé sur les performances de l'EDI. Aussi bien le temps de démarrage que le temps de réponse de l'interface utilisateur ont été amélioré de façon significative. De plus, il y a égalements les modifications suivantes:
  • Certains modules, auparavant disponibles depuis le Centre de Mise à Jour, ont été rajouté dans la distribution standard. Ces modules fournissent plusieurs fonctionnalités, y compris le support pour:
    • débogage de servlets et applications écrites avec la technologie JavaServer PagesTM.
    • création et édition de bibliothèques de tag (tag libraries).
    • monitoring des requêtes HTTP.
    • Paquetage de fichier WAR.
    • Connection aux & navigations des bases de données.
  • Certaines fonctionnalités qui existaient dans les versions précédentes de l'EDI ont été supprimées de la distribution standard:
    • support RMI
    • support JNDI.
    • Éditeur Arborescence XML (à cause de problème de performances - voir issue 31656) Vous pouvez télécharger ce module depuis le Centre de Mise à Jour.
    • Support Scripting.
    • Assistant Nouveau Module.

Modifications Entre les Versions 3.4 et 3.4.1

La version 3.4.1 était à l'origine qu'une version ne fixant que des bogues, mais elle a aussi eu les modifications suivantes:

  • La commande concernant les documentations a été rajoutée au menu View pour permettre la visualisation des documentations Javadoc montées dans un navigateur web.
  • La version supportée de Ant a été la version 1.5.1.

Modifications Entre les Versions 3.3.2 et 3.4

La liste qui suit reprend les modifications qui sont apparues pour la première fois dans l'EDI NetBeans 3.4:

  • Ajout de la barre d'outils dans la fenêtre Éditeur de Source.
  • Ajout d'un outil de maintenance des bases de données pour la completion de code.
  • Support amélioré pour l'intégration de PVCS et VSS dans l'EDI.
  • Ajout du support des fonctionnalitées du JDK version 1.4, comme les assertions.

Configurations Requises

Puisque l'EDI NetBeans est écrit purement en Java, il devrait tourner sur n'importe quelle implémentation de Java 2 SDK, Edition Standard.

Matériel

Les recommendations matérielles sont similaires pour toutes les plateformes, mais elles peuvent différer légèrement dans certains cas. La configuration recommandée pour l'exécution de l'EDI NetBeans est :
  • Espace Disque: 125 Moctets
  • Mémoire: - 256 Moctets
  • Processeur: Processeur PIII/500 (systèmes d'exploitation Windows et Linux), station de travail Ultra 10(système d'exploitation SolarisTM), Processeur Alpha 500MHz (Open VMS) ou équivalent
Remarque: Si la mémoire de votre système est inférieure à la recommendation faite ci-dessus, vous devriez mettre la taille maximale de heap  plus basse dans le fichier bin/ide.cfg. Par exemple, si votre système a 128 Moctets de mémoire, vous pouvez modifier le paramètre -J-Xmx96m  en -J-Xmx48m.

Système d'Exploitation

Tout système d'exploitation supportant le SDK JavaTM 2, Édition Standard. Ci-dessous, la liste des plateformes sur lesquelles l'EDI NetBeans peut s'exécuter. Si vous connaissez d'autres plateforme, faites le nous savoir.
  • Système d'Exploitation Windows 95, 98, NT 4.0, 2000, XP, et ME
  • Système d'Exploitation Solaris 8 et Solaris 9
  • Toutes les distributions Linux
  • OS/2
  • Open VMS 7.2-1 ou supérieur
  • Mac OS X 10.1.1 ou supérieur
  • HP-UX

Logiciel

L'EDI NetBeans requiert une JVM compatible Java 2. Téléchargez la dernière version du JDK approprié (v. 1.3.x ou 1.4.x) depuis les sites suivants :
  • Windows, Solaris, Linux: http://java.sun.com/j2se/1.4/download.html
  • OS/2: http://service.boulder.ibm.com/asd-bin/doc/en_us/catalog.htm
  • Open VMS: http://h18012.www1.hp.com/java/download/index.html
  • Mac OS X: Mac OS X 10.1.1 ou supérieur. Les mises à jour de Mac OS sont disponibles via le mécanisme de mise à jour de logiciels d'Apple.Voir également le paragraphe installation dans ce document. Plus d'informations concernant l'utilisation de l'EDI NetBeans sur Mac OS X est disponible sur le lien https://netbeans.org/kb/articles/mac.html

Si vous tourner sous un système Microsoft Windows, veuillez noter que l'installateur runide.exe ne détecte pas les versions béta des JDK. Vous pouvez définir le paramètre -jdkhome jdk_home_dir dans le fichier ide.cfg si vous désirez utiliser un autre JDK que celui détecté lors de l'installation. Voyez le paragraphe Les Paramètres de l'Exécuteur et de Démarrage pour plus d'information.

Installation

Pour toutes les plateformes vous pouvez télécharger le fichier d'archives .zip, .tar.gz ou .tar.bz2 et le décompresser sur le disque dur en utilisant votre outil favoris. Vous devez ensuite personaliser les paramètres de démarrage pour dire à l'EDI où trouver le SDK. Veuillez lire le paragraphe Les Paramètres de l'Exécuteur et de Démarrage pour les détails.

Sur les plateformes Microsoft Windows, vous pouvez télécharger et exécuter l'installateur .exe qui vous guide à travers les étapes requises.

Certains installateurs exécutables ".bin" sont disponibles pour de nombreuses plateformes UNIX. Vous devez en faire des exécutables avant de les exécuter:

$ chmod +x NetBeans.bin
$ ./NetBeans.bin

Bien que l'installateur recherche toutes les versions de JDK installées, et vous demande quelle version de JDK vous aimeriez utiliser, vous pouvez accélérer le processus d'installation en spécifiant une version de JDK dans la ligne de commande:

$ ./NetBeans.bin -is:javahome <path_to_your_jdk>

Sur Mac OS X

  • Utilisez untar ou unzip pour décompresser NetBeans. Remarque: unzip fait partie des outils de développeurs et n'est donc pas disponible dans une installation normale. Pour décompresser les fichier tar, utiliser gnutar au lieu de tar.
  • Pour lancer l'EDI NetBeans, ouvrez l'application Terminal et allez dans le répertoire netbeans/bin.
  • Lancer l'EDI NetBeans avec le look and feel standard en utilisant la ligne de commande suivante:
    ./runide.sh -jdkhome /Library/Java/Home
    Le paramètre -jdkhome peut être omis si vous avez défini la variable d'environnement JAVA_PATH ou JDK_HOME pointant vers /Library/Java/Home
  • L'EDI NetBeans peut être lancé dans le look and feel Aqua en utilisant:
    ./runide.sh -jdkhome /Library/Java/Home -ui com.apple.mrj.swing.MacLookAndFeel -fontsize 10

    Remarquez que les paramètres sont incompatibles entre les look and feels normal et Aqua. Il vous est donc conseillé d'avoir deux répertoires utilisateurs différents si vous désirez tester les deux look and feels. Utilisez le paramètre -userdir pour sélectionner les différentes répertoires utilisateurs.

Les Paramètres de l'Exécuteur et de Démarrage

L'EDI est lancé par un exécuteur. Des exécuteurs pour plusieurs plateformes sont situés dans le sous-répertoire bin du répertoire d'installation.

Pour UNIX, le Bourne shell script runide.sh est l'exécuteur.
Pour Microsoft Windows, utilisez l'exécutable runide.exe ou runidew.exeexecutable. runide.exe est une application nécessitant la console de Microsoft Windows. Lorsque vous lancer runide.exe, une console s'ouvre sur le bureau avec stderret stdout du EDI NetBeans. Vous pouvez enfoncer les touches Ctrl-Break pour obtenir un thread dump, ou enfoncer les touches Ctrl-C pour quitter le programme. runidew.exe est l'exécutable pour lancer l'EDI NetBeans comme une applications Windows, sans aucune console.
Pour OS/2 runideos2.cmd est l'exécuteur.
Pour OpenVMS runideopenvms.com est l'exécuteur.

L'exécuteur charge la JVM, construit le classpath de l'EDI, le passe, ainsi que d'autres paramètres par défaut, à la JVM, et laisse la JVM lancer l'application Java. Il s'occupe également de redémarrer l'EDI après que vous ayez utiliser le Centre de Mise à Jour.

Vous pouvez passer des paramètres de démarrage à l'exécuteur en utilisant le fichier ${IDE_HOME}/bin/ide.cfg. L'exécuteur essaye de lire ce fichier avant qu'il ne commence à parser les options de la ligne de commande. Vous pouvez mettre les options sur plusieurs lignes.

Les options suivantes sont disponibles:

-h
-help
affiche la description de chaque paramètre de démarrage les plus courants.

-jdkhome jdk_home_dir
utilise la version spécifiée du SDK Java 2 au lieu du SDK par défaut. Par défaut, sur les systèmes Windows, l'exécuteur regarde dans la base de registres de Windows et utilise le dernier SDK disponible.

-classic
utilise la JVM classique, au lieu du client par défaut Java HotSpot.

-cp:p classpath_supplémentaire
ajoute le classpath spécifié devant le classpath de l'EDI. Cette option n'est généralement pas recommandée.

-cp:a classpath_supplémentaire
-cp classpath_supplémentaire
ajoute le classpath spécifié derrière le classpath de l'EDI. Cette option n'est généralement recommandée que pour ajouter les fichiers JARs permettant de personnaliser le Look and Feel de NetBeans, que vous pouvez également ajouter au répertoire  lib/ext/ du EDI NetBeans. Voyez l'aide en ligne pour plus d'information concernant le montage des bibliothèques de développement de l'utilisateur.

-Jparamètre_jvm
passer le paramètre spécifié directement à la JVM.

-ui nom_classe_UI
utilise une classe donnée comme look and feel de l'EDI.

-fontsize taille
utilise une taille donnée en points comme taille de police de base pour l'interface utilisateur de l'EDI..

-locale langage[:pays[:variante]]
utilise la localisation spécifiée.

-userdir répertoire_utilisateur
spécifie explicitement le répertoire utilisateur, qui est l'endroit où sont stoqués les paramètres de l'utilisateur. Si cette option n'est pas utilisée sous UNIX, l'emplacement est ${HOME}/.netbeans/3.5. Sous les systèmes Microsoft Windows, l'endroit par défaut est .netbeans\3.5 en dessous de votre profile Windows (par exemple, c:\Documents and Settings\votre_login).
-J-Dnetbeans.popup.linuxhack=true
sous quelques gestionnaires de fenêtre Linux, cette option fixe le bogue #12496, dans lequel les menus contextuels apparaît dans le coin supérieur gauche de l'écran.
-J-Dnetbeans.tab.close.button.enabled=false
Supprime le bouton fermer des onglets dans l'Éditeur de Source, et d'autres fenêtres.

Compatibilité/Mise à jour

Lorsque vous exécuter pour la première fois l'EDI, vous pouvez importer les paramètres que vous utilisez dans une installation précédente de l'EDI. Ces paramètres inclut les paramètres spécifiques au projet ainsi que les options globales. Si vous choisissez de ne pas importer les paramètres d'une version précédente, l'EDI commence avec des paramètres par défaut.

Si vous utilisez un installateur, l'installateur vous guide à travers les choix d'import de projets. Si vous exécutez l'EDI depuis une archive non compressée, l'Assistant d'Import des Paramètres apparait la première vois que vous lancez l'EDI et vous guide à travers vos choix.

Note:L'installateur ne reconnaît que les installations précédentes où le répertoire utilisateur réside dans l'endroit par défaut. Il ne reconnaît pas les installations où vous avez spécifié un répertoire utilisateur en utilisant le paramètre -userdir. Si vous désirez importer les paramètres d'un EDI que l'installateur ne reconnaît pas, téléchargez une version archive de l'EDI au lieu de l'installateur.

Vous pouvez importer les paramètres depuis l'EDI NetBeans v. 3.0, 3.1, 3.2, 3.3, et 3.4. Vous pouvez également importer les paramètres de l'Édition Communauté de Sun ONE Studio (auparavant ForteTM for Java) versions 2.0, 3.0, et 4.

Dans l'EDI NetBeans, version 3.2 et plus, et Sun ONE Studio, version 3.0 et plus, vous pouvez trouver le répertoire utilisateur lorsque vous exécutez le logiciel. Sélectionnez Help | About, et cliquez sur l'onglet Detail.

Bien qu'il soit possible d'importer les paramètres d'une installation EDI précédente dans l'EDI NetBeans 3.5, il n'est pas possible d'importer tous les paramètres de l'EDI NetBeans 3.5 dans une version précédente de l'EDI.

Problèmes Connus

Voici la liste des principaux problèmes non résolu dans cette version:
  • Description: De nombreux EDI et systèmes plantent à cause d'une mauvaise interaction entre la plateform Java et certains drivers video sous Microsoft Windows. En particulier, il existe des problèmes entre la JDK version 1.4.1_01 et les cartes graphiques ATI Radeon comme indiqué dans Bug Parade issue 4713003.

    Workarounds:
    • Mettre à jour votre driver video.
    • Installer la JDK version 1.4.1_02 ou la JDK version 1.4.2.
    • Désactiver l'utilisation de DirectX par l'EDI. Vous pouvez désactiver DirectX en suivant cette procédure:
      1. Étendre le répertoire d'installation de l'EDI et ouvrir le fichier bin/ide.cfg.
      2. Ajouter le paramètre suivant: 
        -J-Dsun.java2d.noddraw=true
        .
  • 32958 Description: Dans certains cas, lorsqu'il s'exécute sous les systèmes Microsoft Windows XP et que la JDK version 1.4.1 est installée, l'installateur de l'EDI NetBeans ne parvient pas à trouver un version de JDK.

    Workaround: Ré-installer le JDK. Si le problème persiste avec cette version de JDK, essayer une version différente de JDK.

  • 32637, 32682, 32633 Description: Lors de l'exécution sous les versions béta de JDK version 1.4.2, il y a un certain nombre de problèmes avec le look and feel GTK+ qui rend l'EDI généralement inutilisable.

  • 21326 Description: un fichier de recette JAR peut être corrompu si vous basculer de projets ou que vous déplacer des fichiers ou des systèmes de fichiers auquel se référait le fichier de recette JAR.

  • 32962 Description: L'EDI vous permet de choisir une classe principale pour le projet qui n'a pas été ajouté au projet, mais il ne la traitera pas comme classe principale.

    Solution: Cliquez-droit sur le fichier et choisissez Tools | Add to Project.

  • 25934 Description: la commande Goto Declaration (Alt-G) fonctionne quelque fois très lentement sur certains systèmes.

  • 11637 Description: Lors de l'utilisation de l'outil de gestion d'import sur des classes internes (inner classes), the name of the outer class is stripped from the reference to the inner class.

  • 28004 Description: Points d'arrêt qui sont mis entre les tags <% et %> d'un JSP ne sont pas mappées avec les lignes correctes dans la servlette.

    Workaround partiel: Dans votre code, ne mettez pas les tags <% et %> sur leur propre ligne. Tout points d'arrêt que vous mettrez sur ces lignes seront correctement mappées. Cependant, si vous mettez les points d'arrêt sur des lignes entre les tags de fin et de fermeture, le point d'arrêt sera mappé à la ligne de servlette, correspondant à la ligne du <% dans le JSP.

  • 23135 Description: Si un module web contient des pages JSP qui ont le même nom mais sont stoqués dans des répertoires différents, alors une erreur de compilation survient lorsque la commande "Build All" est utilisée. Ce problème se produit parce que le compilateur JSP de Tomcat crée deux servlettes qui on le même pathname.

    Workaround: Soit vous compiler les pages JSP une par une, soit vous mettez le mode d'intégration de l'EDI au minimum. Voyez la page d'aide du module Tomcat intitulée Adding a Tomcat 4.0 Installation pour plus d'informations concernant le parametrage du mode d'intégration de l'EDI.

  • 29657 Description: Module web valide qui contient des classes produites par l'éditeur visuel de l'EDI ne compile pas.

    Workarounds (faites en un seul ):
    • Sur l'onglet Runtime tab, étendez le noeud Server Registry et naviguez jusqu'au noeud pour l'instance de Tomcat appropriée. Mettez la propriété Mode d'Intégration de l'EDI sur Minimal.
    • Installer la JDK version 1.4.2, lorsqu'il sera sorti.


  • 24443 Description: Lors du débogage d'une servlette, si vous rafraichissez votre navigateur plusieurs fois dans une succession rapide, une exception peut se produire.

    Workaround: Si l'exception se produit, redémarrez le serveur web avant de faire un resume du débogage.

  • 20384 Description: JARs de bibliothèque de Tags ne peut être remplacé dans le répertoire WEB-INF/libaprès avoir parsé et exécuté les JSP dans le module web approprié. Le parseur JSP, compilateur, et/ou exécuteur retient quelques fois des liens sur les JAR de bibiothèque de Tags dans le répertoire WEB-INF/lib. Sous les systèmes Windows, le résultat est que les JAR ne peuvent être effacé, et cela de façon intermittente, résultant en une exception "Cannot delete file...". Sous le système d'exploitation Solaris, les JARs peuvent être supprimés, mais lorsqu'ils sont remplacés par de nouveaux JARs, le compilateur "voit" toujours les anciennes versions des JARs. Pour obtenir le fichier JAR correct, vous devez redémarrer l'EDI.

    Workaround: Développer les bibliothèques de Tags en référencant les TLD et classes de gestionnaire de tag directement, sans mettre la bibiothèque de tags dans un fichier JAR jusqu'à la fin du cycle de développement.
    1. Création de la bibiothèque de Tag dans le répertoire WEB-INF/lib et faire tout le développement là.
    2. Set the tag handler generation root to the web module's WEB-INF/classes directory.
    3. Utiliser la bibliothèque de Tags depuis une page JSP, soit réferencer directement à l'URI de la taglib comme /WEB-INF/lib/mytaglib.tld ou créer un alias dans le descripteur de déployement avec une entrée de bibliothèque de tags.

Elle apparaîtra dans le fichier de log du serveur et/ou dans la fenêtre du navigateur web. Ce bogue est la conséquence du bogue Tomcat 6907.

    Workaround: Supprimer la copie de xerxes.jar de votre répertoire ${JDK_HOME}/jre/lib/ext.

  • 26324 Description: Lorsque vous exécutez l'EDI sous JDK version 1.4 et que vous utilisez le look and feel Windows, la recherhce  dans la fenêtre d'Aide obscure le texte mis en évidence.

    Workaround: Utilisez une version different du JDK ou un autre look and feel.

  • 32019 Description: Il n'est pas possible de copier/coller des containeurs qui utilisent un gestionnaire de layout personnalisé.

    Workaround: Mettre le gestionnaire de layout du containeur à un layout standard (comme FlowLayout). Ensuite, faites le copier/coller, et remettre le gestionnaire de layout correct.

  • 29614 Description: Quelques fois, l'EDI pour prendre un long moment pour traiter des Transformations XSL. Pour se prémunir d'une surcharge de la JVM, la commande de Transformation XSL est désactivée lorsqu'une autre transformation a lieu.

    Note: Plusieurs processeurs XSLT différents sont disponibles. Par défaut, l'EDI utilise le parseur XALAN, mais vous pouvez trouver qu'un autre parseur accomplit mieux son travail pour les transformations que vous effectuez régulièrement. Si vous désirez utiliser un autre parseur, consultez les spécifications de JAXP 1.1 ou voyez la FAQ située sur la  page d'accueil du module NetBeans XML.

  • 22802 Description: L'EDI peut se bloquer lors de l'Explorateur de la Base de Données pour vous connecter à une base de données sous certaines circonstances, comme une surcharge de réseau, temps de réponse lent du seveur de base de données, ou le rapatriement d'un grand nombre de données.

  • 32093 Description: Si vous accéder au Web via un proxy, l'utilisation du Gestionnaire Javadoc pour monter Javadoc comme système de fichiers HTTP peut ne pas travailler.

Utilisez le système Issuezillapour vérifier les bogues actuellements ouverts. Ou vous pouvez trouver la version la plus récente de cette liste à l'adresse https://netbeans.org/community/releases/35/README_fr.html#KnownProblems.
